# План портирования BloomFS: Go → Rust

> Версия документа: 1.0 · Дата: 2026-06-20 · Статус: **выполнен**
> Архитектура и on-disk формат **заморожены** (см. [SPEC.md](SPEC.md) §E10,
> changelog 1.26). Этот документ описывает выполненный перенос Go-прототипа на
> Rust (11 крейтов, ~6 200 строк, `rust/`). Читается без обращения к чату.
>
> **Changelog 1.0:** первый план. Порядок слоёв снизу вверх по зависимостям;
> Go-прототип как исполняемый оракул; раскол bloom-библиотеки на критический
> минимум (5a) и полный standalone-crate (5b); сквозные решения (конкурентность,
> ошибки, сериализация, BLAKE3, паритет формата); тестовая стратегия; бэклог
> отложенных оптимизаций; риски.

## Принцип

- **Снизу вверх по зависимостям.** Каждый слой переносится со своими тестами;
  следующий слой не начинаем, пока текущий не зелёный.
- **Go-прототип = исполняемый оракул.** Кодеки сверяем байт-в-байт через
  golden-векторы — там, где не задействован хеш (inode, bitmap, uberblock, DDT,
  dir-страницы). Хеш-зависимое (dedup-ключи) проверяем на самосогласованность.
- **Никакого big-bang.** `fs` — самый объёмный слой; рвём его на уже существующие
  под-зоны и переносим инкрементально.

## Bloom-библиотека: раскол 5a / 5b

`xxh3-bloom` — наша библиотека (отдельный репозиторий). `dir` использует у фильтра
**ровно три вызова**: `NewBlockedTuned(cap, fp)`, `.Add(name)`, `.Test(name) bool`.
Фильтр **не персистится** — он пересобирается из индекса (`NewBlockedTuned` + `Add`
по каждому имени) на mount и в `rebuildFilter`. Поэтому:

- **5a — критический путь BloomFS, маленький.** Перенести только: конструктор с
  tuning-математикой (`minBlocksForK` / `EstimateBlockedFP`), `blockOffset`
  (fastrange-выбор блока + split хеша на h1/h2), `Add`, `Test`, на xxh3. Без
  ассемблера, без сериализации, без `Hasher`-абстракции (хардкодим xxh3). Это
  разблокирует `dir`.
- **5b — полный standalone-crate, вне критического пути.** Сюда уходит тяжёлое и
  Rust-расходящееся: `SimdFilter` с рукописным AVX2 (`filter_simd_amd64.s`) →
  `std::arch::x86_64` intrinsics (или `std::simd`) со scalar-фоллбэком;
  `AddBatch` / `TestBatch`; prefetch-asm; формат `'BBLM'` (`WriteTo` / `ReadFrom` /
  `MarshalBinary` / `Equal`). Самостоятельная ценность, портируется параллельно,
  **не блокирует** BloomFS.

**Паритет математики, не байтов.** Раз фильтр не пишется на диск, формат `'BBLM'`
**не входит** в замороженный on-disk контракт BloomFS. Rust-фильтру нужно
воспроизвести те же `k`, `numBlocks`, fastrange и split хеша, чтобы FP держался
≈0.01 на cap (тогда сохраняется оценка дрейфа ~1.25× cap из deferred-rebuild).
Бит-в-бит совпадать с Go-фильтром не требуется.

## Сквозные решения (НЕ прямой перенос)

- **Конкурентность.** Go `RWMutex` → старт на `parking_lot::RwLock` для паритета.
  Тот же reader-contention никуда не делся — реальные ускорения (шардинг индекса,
  lock-free чтение) теперь настоящая работа, не «оптимизация на потом». `-race`
  исчезает (владение даёт это на компиляции); для lock-free структур — `loom`,
  для UB — `miri`.
- **Сериализация — ручной little-endian, НЕ serde.** Формат заморожен байт-в-байт;
  `MarshalInto(dst)` / `MarshalLen()` → запись в `&mut [u8]`. Переиспользуемый
  `metaBuf` в Rust естественен (один `Vec<u8>`), но дисциплина «писать каждый байт
  явно» (reserved-байты инода, флаг `Raw` в dedup) остаётся — reuse-буфер так же
  тащит мусор в незаписанных байтах.
- **Ошибки.** Go sentinel-errors → enum'ы (`thiserror`); маппинг в errno живёт в
  `fusefs`, как сейчас.
- **BLAKE2b → BLAKE3 меняет dedup-хеш ⇒ образы несовместимы между Go и Rust.**
  Решение: **fresh-format only**, кросс-реализационная совместимость образов не
  цель (у прототипа и так нет on-disk гарантий). Golden-векторы сверяем только на
  не-хеш-кодеках; хеш-зависимое — на самосогласованность.
- **FUSE.** `fuser` ≠ go-fuse по форме (callback + Request/Reply вместо интерфейсов
  узлов), но свойство «тонкий шим без логики хранения» сохраняется.

## Durability-контракт (наследуется портом)

Из [SPEC.md](SPEC.md) §E10 — порт наследует **как есть**:
коммит привязан только к явному `fsync` приложения и чистому unmount; фонового
периодического коммита нет (окно потери по времени не ограничено); один fsync
коммитит весь снимок атомарно (сильнее POSIX, ZFS-style). Опциональный фоновый
коммит (jbd2-критерий) — в бэклоге ниже, не в паритете.

## Тестовая стратегия

- **crash-injection** тесты переносятся напрямую (они на уровне `Device`).
- **round-trip кодеков + инварианты `dir`** → `proptest` / `cargo-fuzz`.
- **бенч-лестница:** `criterion` (микро) → file-image → **fio поверх реального
  маунта** (незакрытая верхняя ступень — закрыть уже в Rust).
- **golden-векторы** из Go-оракула для не-хеш-кодеков.

## Бэклог отложенных оптимизаций (ПОСЛЕ паритета и fio)

Это уже не «потом в Rust», а явные пункты после функционального паритета:

1. open-addressing хеш-таблица для индекса `dir` (swizzling — в Rust контроль
   layout реален);
2. фоновый коммит по jbd2-критерию (если решим ограничить окно потери);
3. dirty-block CoW метаданных (перф, рискованно — только после паритета);
4. шардинг блокировок / масштабирование чтения;
5. полный crate `xxh3-bloom` 5b (SIMD/batch/serde) — если нужен standalone.

Ни один пункт не начинаем раньше, чем порт пройдёт fio на реальном маунте.

## Главные риски

- **`fs` (3135 строк) — основной объём.** Рвём по уже выделенным под-зонам: lookup,
  write-path, truncate/sparse, timestamps, handles/open-count, readdir-snapshot,
  permissions. Каждую — с её posix-тестом зелёным.
- **5a на критическом пути `dir`.** Без scalar-фильтра `dir` не собирается —
  переносим его одним из первых; математику FP воспроизводим точно.
- **`fuser` форма callback'ов** отличается от go-fuse — заложить время на адаптацию
  биндинга (логика хранения не меняется, меняется только форма шима).
