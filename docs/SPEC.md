# Техническое задание: высокоскоростная крипто-дедуплицирующая файловая система «BloomFS»

> Версия документа: 1.25 · Дата: 2026-06-20 · Статус: черновик к согласованию
> Название: **BloomFS** (утверждено). Имя ведёт от уникального дифференциатора —
> сегментированного bloom-поиска по каталогу (этого нет ни в ZFS, ни в btrfs, ни в
> ext4). Вариант BTZFS отклонён: внутри сидит «ZFS» (читается как форк) и «Zip» ≠
> реальный компрессор ZSTD/LZ4.
> Документ синтезирован из проектного мозгового штурма и самодостаточен:
> читается без обращения к исходному чату.

> **Changelog 1.25:** зеро-аллок снимок метаданных в `cow.Commit` (устранение
> GC-давления на коммите). Раньше каждый коммит аллоцировал свежие слайсы под
> сериализацию bitmap+ddt+inode-таблицы, причём `inode.MarshalBinary` делал
> `make([]byte,128)` **на каждый инод** — на каталоге в 4000 файлов это ~4047
> аллокаций / 1.9 MB **на коммит**, и линейно росло бы с числом инодов (на 100k
> файлов GC задавил бы коммит). Теперь все три структуры пишутся методами
> `MarshalInto(dst)`/`MarshalLen()` в **переиспользуемый буфер** (`FS.metaBuf`,
> размер метаслота, аллоцируется один раз на `Mount`); старый `Marshal()` — тонкая
> обёртка, формат на диске байт-в-байт прежний. Байты, прежде полагавшиеся на
> zero-init свежего буфера (3 reserved-байта инода, флаг `Raw` в dedup), пишутся
> явно — корректность при переиспользовании буфера; хвост за суммарной длиной не
> читается на `Mount` (длины секций в уберблоке). Итог `BenchmarkCreateUnlinkCommit`:
> 4047→**41** аллокаций/оп, 1.9 MB→**94 KB**, 358→**229** мкс (mem); 517→**380**
> мкс на файле-образе. Главное — стоимость коммита по аллокациям теперь **плоская
> независимо от числа инодов**. Это **только аллокации**: дисковая запись по-прежнему
> переписывает весь метаслот (ping-pong целиком), а её цена на реальном диске
> упирается в `fsync`; уменьшение самой записи (dirty-block CoW метаданных) — это
> отдельная архитектурная переработка, сознательно отложена (риск перед заморозкой
> под Rust), а снижение `fsync` — это батчинг коммитов (отдельное durability-решение).

> подтверждено микробенчами `fs/bench_test.go`). (1) **Постраничная раскладка каталога:**

> `cap/4` удалений (счётчик `tomb`), амортизируя стоимость до O(1) на unlink при

## 0. Назначение и цели

### 0.1. Назначение

BloomFS — userspace-файловая система с прицелом на **мгновенный поиск**,
**параллельное асинхронное чтение** и **устойчивость к фрагментации**, поверх которых
работают **прозрачное сжатие, шифрование и блочная дедупликация**. Целевые сценарии:
хранилище с большим числом файлов в одной директории, где классические FAT/ext2 деградируют.

### 0.2. Ключевая идея

Связка **xxhash + фильтр Блума + сегментация каталога по 4000 файлов** даёт поиск,
упирающийся в кэш процессора, а разделение «имя ↔ inode» + блочный конвейер
(сжатие → дедуп → шифрование) даёт компактность и безопасность «из коробки».

### 0.3. Нефункциональные требования (NFR)

| # | Требование | Цель | Как достигается |
|---|------------|------|-----------------|
| NFR-1 | Скорость поиска в каталоге | ~14–22 нс (in-memory индекс) | xxhash64 + фильтр Блума + map/B+-дерево |
| NFR-2 | Деградация при росте числа файлов | отсутствует | сегменты ≤ 4000 файлов, у каждого свой фильтр |
| NFR-3 | Параллельное асинхронное чтение | без глобальной блокировки | гранулярные RWMutex/RCU, карта блоков в inode |
| NFR-4 | Устойчивость к дефрагментации | «пофиг» | кластер аллокации 4К, экстенты для архивов, блочное дерево для random |
| NFR-5 | Прозрачное сжатие | ZSTD/LZ4, по recordsize | переменный логический блок 4–32К поверх кластера 4К (§4.5) |
| NFR-6 | Прозрачное шифрование | AES-XTS | tweak = адрес физического кластера |
| NFR-7 | Дедупликация | по контент-хешу несжатого блока | DedupTable + RefCount; короткое замыкание до сжатия (§5) |
| NFR-8 | Пользователи/группы | до 256 (1 байт) | micro-POSIX права в 3 байта |

### 0.4. Важная оговорка о цифрах 14–22 нс

Замеренные **14 нс (Rust) / 22 нс (Go)** — это латентность поиска по **in-memory
индексу** (фильтр Блума + map). Это не сквозная латентность файловой операции.
Сквозной путь через FUSE включает переключения контекста ядро↔userspace и будет на
**порядки выше** (микросекунды). См. **§B10 (FUSE-потолок)** — это не повод
отказываться от быстрого индекса, но цифры в маркетинге надо подавать честно.

## 1. Стек технологий и среда исполнения

- **Среда:** userspace через **FUSE** (Filesystem in Userspace) — никакого кода в
  пространстве ядра на этапе разработки (баг в ядре = синий экран / kernel panic).
- **Языки:** **Go (1.20+)** для R&D-прототипа → **Rust** для production-релиза.
  Go быстро отсекает ложные архитектурные пути; Rust даёт безопасность памяти и
  предсказуемость без пауз GC.
- **Эмуляция диска:** на старте — один файл-образ фиксированного размера, к которому
  относимся как к массиву блоков по **4096 байт**. Никакой привязки к реальному
  накопителю в прототипе.

### 1.1. Маппинг концепций Go → Rust (для будущего порта)

| Концепция в Go | Эквивалент в Rust | Выигрыш |
|----------------|-------------------|---------|
| `go func()` (горутины) | `tokio::spawn` (async tasks) | zero-cost, нет рантайма Go |
| `chan` (каналы) | `tokio::sync::mpsc` / `crossbeam` | максимальная скорость обмена |
| `interface` (`BlockDevice`) | `pub trait BlockDevice` | статическая диспетчеризация |
| `sync.RWMutex` | `tokio::sync::RwLock` | async-блокировки, не вешают поток ОС |
| `map[[32]byte]…` | `HashMap<[u8;32], …>` | то же поведение |

### 1.2. Интерфейс блочного устройства

```go
type BlockDevice interface {
    ReadBlock(blockNum uint64) ([]byte, error)
    WriteBlock(blockNum uint64, data []byte) error
}
```

Реализации: (1) in-memory `[]byte` для прототипа; (2) файл-образ через `os.OpenFile`
с флагом `O_DIRECT` (обход кэша ОС — **с оговоркой §B11**: понадобится собственный
буферный кэш).

### 1.3. Внешние библиотеки (готовые «плечи гигантов»)

| Назначение | Go | Rust |
|------------|----|----|
| Хеш имён (быстрый, не крипто) | `cespare/xxhash/v2` | `xxhash-rust` / `twox-hash` |
| Хеш для дедупа (крипто) | `zeebo/blake3` | `blake3` |
| Сжатие | `klauspost/compress/zstd`, `lz4` | `zstd`, `lz4_flex` |
| Шифрование | `crypto/aes` (XTS) | `aes`, `xts-mode` |
| Фильтр Блума | `bits-and-blooms/bloom/v3` | `fastbloom` / собственный |
| B+-дерево (референс) | `bbolt` (mmap B+-tree) | движки sled / собственный |
| FUSE-мост | `winfsp/cgofuse`, `bazil.org/fuse` | `fuser` |

## 2. Разметка диска (On-Disk Layout)

Виртуальный диск (файл-образ) логически разбит на блоки по 4 КБ:

```
┌──────────────┬───────────────┬───────────────┬──────────────┬────────────────────┐
│  Суперблок   │  Таблица инод │  Bitmap инод  │ Bitmap блоков│   Блоки данных     │
│  (блок 0,    │  (32 inode    │  (свободные   │ (свободные   │ (сжатые+зашифров.  │
│  не шифр.)   │   на блок)    │   иноды)      │  блоки)      │  независимые куски)│
└──────────────┴───────────────┴───────────────┴──────────────┴────────────────────┘
```

- **Суперблок** — первые байты диска. Хранит: размер блока, общее число блоков,
  указатель на корневой каталог, **версию формата** (задел на совместимость),
  параметры шифрования (соль/идентификатор схемы, но не ключи). **Не шифруется** —
  иначе ФС не поймёт, как себя примонтировать.
- **Таблица инод** — массив 128-байтных inode (см. §4); 32 inode в один блок 4 КБ.
- **Bitmap инод / блоков** — битовые карты занятости (см. §B2, §B5).
- **Блоки данных** — независимые «слепые кубики» (см. §4.4 — почему слепые).

## 3. Подсистема каталогов — «мгновенный поиск»

### 3.1. Виртуальные сегменты

Логическая папка = **связный список виртуальных сегментов**, каждый вмещает строго
**до 4000 файлов**. Переполнился — создаётся следующий сегмент-«продолжение».

Почему это летает:
- Фильтр Блума остаётся крошечным и точным (на 100 000 файлов в одном фильтре он бы
  раздулся / захлёбывался ложными срабатываниями).
- Перестройка фильтра при мутации — это микросекунды на 4000 элементов, а не на
  сотни тысяч.
- Локализация кэша: работаем только с одним сегментом, а не со всей гигантской папкой.

```go
type VirtualSegment struct {
    Filter *bloom.BloomFilter  // ~4 КБ, 1% FP, в RAM
    Index  map[uint64]InodeID  // ключ: xxhash(name) → InodeID
}
type LogicalDirectory struct {
    InodeID  InodeID
    Segments []*VirtualSegment  // связный список «продолжений»
}
```

### 3.2. Конвейер поиска файла

```
                  [Запрос файла "report.txt"]
                               │
                               ▼
                  hash = xxhash.Sum64("report.txt")
                               │
              ┌──── по сегментам (до ~100 проверок) ────┐
              ▼                                          │
       [Фильтр Блума сегмента]                           │
              │                                          │
      ┌───────┴────────┐                                 │
 (бит = 0 → НЕТ)  (все 1 → ВОЗМОЖНО)                      │
      │                │                                 │
   след. сегмент   [map/B+-дерево: hash → InodeID]        │
                       │                                 │
                  [сверка строки имени] ── защита от коллизий xxhash
                       │
                   [InodeID]  ──или──  [ENOENT, 0 чтений с диска]
```

> Схема показывает линейный обход сегментов (прототип / случай одного сегмента). При
> **>1 сегмента** обход заменяется прямой хеш-адресацией `xxhash(name) % N` → ровно
> один сегмент (§3.5).

### 3.3. Мутация каталога (add/delete)

Фильтр Блума **не модифицируется** (классический Блум не умеет удалять — общие биты
сломают другие файлы). Вместо этого:
1. Меняем индекс сегмента (map/B+-дерево).
2. **Перестраиваем фильтр с нуля** одним проходом по актуальным ключам сегмента (он
   уже в кэше) — за микросекунды.
3. **Атомарно подменяем** указатель на фильтр (`atomic.Pointer` в Go). Старые читатели
   спокойно дочитывают прежнюю версию.

Это даёт идеальную плотность фильтра без «призраков» от удалённых файлов и без
накладных расходов counting/cuckoo-фильтров.

**Батчинг при массовом удалении (урок из стадии А).** Перестройка одного фильтра —
действительно микросекунды, но **пер-операционный** rebuild на массовом `rm`
(удалили N файлов из сегмента) даёт **O(N²)** (каждое удаление пересобирает весь
фильтр). Поэтому: при пакетном удалении/перемещении изменения в индексе применяются
сразу, а **rebuild фильтра откладывается до конца пачки** (один проход вместо N).
Реализационно — флаг `dirty` на сегменте + ленивая перестройка перед первым `Test`
после пачки, либо явный `Flush()` в конце транзакции (естественно ложится на
CoW-транзакции, §B1). Подтверждено бенчмарком: наивный вариант на 5000 удалений шёл
десятки секунд именно из-за повторных rebuild.

**Реализовано (changelog 1.24): амортизированная перестройка через счётчик tombstone'ов.**
`remove` не трогает фильтр вовсе — запись выбывает только из индекса, а её биты остаются
в фильтре «протухшими». Это безопасно: фильтр здесь чисто оптимизация, и `find` сверяет
каждый Bloom-хит с индексом, так что лишний выставленный бит = всего лишь редкий
false-positive, а не ошибочный хит. Полный `rebuildFilter` откладывается до накопления
порога удалений (`cap/4`, счётчик `tomb` на сегменте), после чего фильтр пересобирается с
нуля и `tomb` обнуляется. Так дорогая перетюнинг-математика конструктора амортизируется до
O(1) на удаление (вместо O(N) на каждое), а дрейф FP ограничен сверху (фильтр ёмкости
`cap` держит в момент rebuild не более ~1.25× ключей). Это покрывает и одиночный churn
(многократный create/unlink одного имени), и массовое удаление — без отдельного `Flush()`.

### 3.4. Хранение ключей в каталоге

В индексе храним не строки (разной длины, неудобно сравнивать), а фиксированный
`xxhash64` имени — идеальный ключ. Запись каталога:

```go
type DirectoryEntry struct {
    NameHash uint64   // ключ для поиска
    InodeID  uint64   // ссылка на метаданные
    NameLen  uint16   // длина имени
    Name     string   // само имя — для листинга и сверки коллизий
}
```

### 3.5. Адресация сегментов: хеш-распределение (ДЕФОЛТ при >1 сегмента)

**Решение (по данным свипа стадии А): для каталога из >1 сегмента поиск идёт через
хеш-распределение, а не линейным сканом списка.**

`segment_index = xxhash(name) % N` → запрос сразу попадает в **единственный** сегмент,
где может лежать файл. Один Bloom-Test + один индекс, независимо от числа сегментов.

Почему это поднято из «при необходимости» в дефолт — замеры на стадии А (16000 файлов):

| сегментов | miss-латентность | вывод |
|---|---|---|
| 1 | 32 нс | базовый cache-miss |
| 4 | 111 нс | линейный скан: +~26 нс/сегмент |
| 16 | 417 нс | деградация ∝ числу сегментов |

Линейный скан добавляет ~26 нс (один cache-miss) **на каждый** сегмент — это и есть
узкое место, а вовсе не размер сегмента. Свип также показал: **общая память не зависит
от размера сегмента** (определяется только FP), значит размер сегмента — чистый knob
латентности/мутации. Хеш-адресация снимает скан-цену и делает оптимальным **мелкий**
сегмент (cap 4000): дешёвый rebuild фильтра (§3.3), дешёвая мутация.

- **Связный список** остаётся тривиальным частным случаем **одного** сегмента (каталог
  ≤ 4000 файлов — целевой NFR-2): там адресация не нужна, латентность ≈ single-segment.
- **Цена хеш-адресации:** при переполнении всех сегментов число `N` удваивается и файлы
  перераспределяются (**extensible / consistent hashing**). Это принятая стоимость, не
  «преждевременная оптимизация» — без неё многосегментный каталог не масштабируется.
- **Статус кода:** прототип стадии А пока использует линейный скан (достаточно для ≤1
  сегмента); переход на хеш-адресацию — задача стадии, где появляются реально
  многосегментные каталоги.

**Параметры по умолчанию (зафиксированы свипом):** FP = **0.01** (~1.3 байта/файл;
0.001 удваивает память зря, 0.05 даёт 5% проваливаний в индекс), capacity сегмента =
**4000**.

## 4. Метаданные файлов — Inode

### 4.1. Размер и поля (ровно 128 байт)

**Решение: inode = 128 байт** (был черновой вариант 64 байта). Причины:
- **32 inode помещаются в один блок 4 КБ** — по-прежнему отличная локальность, одним
  чтением блока — пачка файлов.
- **128 байт = ровно кэш-линия Apple Silicon** (там 128 Б; на x86 — 64 Б, т.е. 2 линии —
  тоже нормально). Кэш-линия платформозависима — 128 покрывает обе архитектуры.
- В 64 байта **не влезали** обязательные поля: три POSIX-таймстампа, `nlink`,
  checksum для bit-rot (§B13), recordsize (§4.5). 128 байт даёт запас + место под
  **инлайновые экстенты**.

| Поле | Тип | Байт | Назначение |
|------|-----|------|------------|
| Size | `uint64` | 8 | размер файла |
| Nlink | `uint32` | 4 | счётчик hard links (§4.2) |
| Generation | `uint32` | 4 | поколение inode (переиспользование номера, NFS-хэндлы) |
| UID | `uint32` | 4 | владелец (реальные POSIX-uid, не помещаются в u8) |
| GID | `uint32` | 4 | группа |
| Atime | `uint64` | 8 | время доступа (по умолчанию relatime, §B16) |
| Mtime | `uint64` | 8 | время изменения данных |
| Ctime | `uint64` | 8 | время изменения метаданных |
| Mode | `uint16` | 2 | биты прав/setuid/sticky (тип — в Type); 0o7777 не влезает в u8 |
| Type | `uint8` | 1 | тип (файл/каталог/симлинк) |
| RecordSizeLog2 | `uint8` | 1 | log2(recordsize), 12..15 = 4..32К (§4.5) |
| Flags | `uint8` | 1 | inline-extents / сжат / зашифрован и т.п. |
| _reserved | `[3]byte` | 3 | выравнивание/задел |
| Checksum | `[8]byte` | 8 | контрольная сумма содержимого для bit-rot (§B13) |
| InlineExtents / BlocksInfo | `[64]byte` | 64 | до ~4 инлайновых экстентов **или** указатель на карту блоков (§4.4) |

> **Номер inode не хранится в самой структуре** — он равен индексу записи в таблице
> инод (экономия 8 байт «бесплатно»).
>
> **Инлайновые экстенты:** маленькие файлы (а их большинство) хранят карту блоков
> прямо в inode и **не требуют отдельного индексного блока** → на одно чтение меньше.
> Флаг в `Flags` различает «inline» и «указатель на внешнюю карту».

```rust
// Rust: статическая проверка размера на этапе компиляции
const _: () = assert!(std::mem::size_of::<Inode>() == 128);
```

> Держимся степеней двойки (64/128/256), не 96 — иначе inode уродливо пересекают
> границы кэш-линий. 128 делит 4096 нацело. Если когда-нибудь не хватит и под 128
> (длинные xattr, много экстентов) — выносим в overflow-блок по указателю (§B16),
> не раздувая «горячий» inode.

### 4.2. Inode не хранит имя

Inode — это «сухие» метаданные (размер, права, адреса блоков). Связь **имя → inode**
живёт **только в каталоге** (телефонный справочник: имя → номер inode). Следствия:
- **Rename за O(1):** меняем ключ в каталоге, сам файл на диске не трогаем.
- **Hard links:** несколько имён в разных каталогах → один InodeID.
- **Безопасное удаление открытых файлов:** `rm` убирает имя из каталога, а inode и
  блоки живут, пока процесс держит файл открытым.

Создание файла = два независимых шага: (1) выделить inode из bitmap; (2) залинковать
`xxhash(name) → InodeID` в текущий сегмент каталога.

### 4.3. Micro-POSIX права (3 байта)

UID (1 байт) + GID (1 байт) + Mode (1 байт). Проверка — одна процессорная инструкция
по битовой маске:

```go
func CanRead(inodeUID, inodeGID uint8, mode uint16, procUID, procGID uint8) bool {
    if procUID == 0 { return true }                       // root
    if procUID == inodeUID { return mode&0o400 != 0 }     // владелец
    if procGID == inodeGID { return mode&0o040 != 0 }     // группа
    return mode&0o004 != 0                                 // остальные
}
```

> Если нужна полная POSIX-совместимость `rwxrwxrwx` (9 бит), Mode расширяется до
> `uint16` — всё равно мизер. Имена пользователей ФС не хранит, оперирует числами;
> UID/GID приходят от ядра через FUSE.

### 4.4. Карта блоков: почему порядок — в inode, а не в самих блоках

**Нельзя** хранить «следующий блок» внутри блока (как цепочка FAT):
- Убивает параллелизм: чтобы прочитать 10-й блок, пришлось бы пройти первые 9.
- Ломает дедупликацию: блок с указателем на «следующий» нельзя переиспользовать в
  другом файле. Блок данных должен быть **абсолютно слепым и независимым**.

Порядок и адреса — в метаданных. Две схемы:

| Схема | Когда | Суть |
|-------|-------|------|
| **Extents (экстенты)** | read-heavy, архивы, длительное хранение | `{старт: блок 1000, длина: 10000}` — пара байт, ноль фрагментации, последовательное чтение |
| **Индексное дерево** | write-heavy, random-доступ, логи, кэши | `BlocksInfo` → индексный блок с массивом `uint64`-адресов; плевать на дыры |

**Гибрид (ext4-стиль):** по умолчанию файл — экстенты (экономия места, быстрое
чтение); при активной хаотичной перезаписи ФС конвертирует карту в индексное дерево.
Для прототипа проще начать **с индексного дерева** (обычный слайс адресов, дружит с
дедупом).

Параллельное чтение: имея полную карту блоков, можно одновременно запустить N
async-потоков, читающих разные блоки (2-й, 5-й, 10-й…) — им не нужно ждать друг друга.

### 4.5. Переменный размер блока (recordsize) поверх кластера 4 КБ

Чтобы обойти «малый кластер» (4К плохо жмётся) без потери компактности, **разделены
две гранулярности**:

- **Кластер аллокации = 4 КБ** (фиксированный) — единица bitmap, адресации и
  выравнивания. Внутренняя фрагментация минимальна.
- **Логический блок (recordsize) = кластер × коэффициент склейки {1, 2, 4, 8}** =
  **{4, 8, 16, 32} КБ**. Только степени двойки. Это единица сжатия **и** дедупа.
  Сжатый логический блок округляется вверх до целого числа кластеров:

```
логический блок 32К → ZSTD → 11К → округление вверх → 3×4К = 12К на диске
экстент: {старт_кластер, длина=3 кластера, лог_размер=32К, сжатый=11К, recordsize_log2=15}
```

Так получаем сжатие на большом контексте (32К жмётся куда лучше 4К) при аллокации по
4К (почти нет потерь на «хвостах»). Поле под `log2` (1 байт) оставляет дверь на
64/128К (log2 16/17); реально включены 4–32К.

**recordsize — свойство файла, детерминированно выводимое из финального размера**
(не из паттерна записи — иначе две копии одного файла могли бы получить разный
recordsize и **не дедуплицироваться**):

| Финальный размер файла | recordsize | log2 |
|---|---|---|
| < 32 КБ | 4 КБ | 12 |
| 32 КБ – 256 КБ | 8 КБ | 13 |
| 256 КБ – 2 МБ | 16 КБ | 14 |
| ≥ 2 МБ | 32 КБ | 15 |

Пороги — настраиваемая константа.

**Когда recordsize фиксируется:**
- Размер известен заранее (`fallocate`/потоковая `cp`) → ставится сразу.
- Последовательная запись с закрытием → **замораживается при первом `close`** по
  достигнутому размеру; старые блоки не переписываются.
- Файл, который растят инкрементально годами (лог, БД), остаётся в мелком recordsize —
  это ровно random/append-нагрузка, которой большой блок не нужен и которая не цель
  дедупа.

**Следствие (что и требовалось):** две копии одного файла → одинаковый финальный
размер → одинаковый recordsize → одинаковое выравнивание блоков → дедуплицируются
полностью. См. §B9 — почему крупный recordsize **не** ухудшает дедуп на истинных
копиях (и его границу применимости).

**Random-write штраф:** перезапись части сжатого 32К-блока = read-modify-write всего
блока (decompress→modify→recompress→rewrite). Поэтому крупный recordsize — только для
архивных/«написал один раз» файлов; random-файлы остаются на 4–8К (см. таблицу выше +
конвертация в индексное дерево, §4.4).

## 5. Конвейер данных: сжатие → дедуп → шифрование

### 5.1. Строгий порядок

```
Логич. блок → Хеш(BLAKE3 несжатого) → дедуп-проверка
   ХИТ  → RefCount++ (НЕ сжимаем, НЕ шифруем, НЕ пишем)
   МИСС → Сжатие(ZSTD/LZ4) → Шифрование(AES-XTS) → Запись → Checksum(шифртекста)
```

Два правила:

1. **Сначала сжатие, потом шифрование.** Зашифрованные данные — белый шум, а шум не
   сжимается ничем. Зашифруешь раньше — размер вырастет.
2. **Дедуп-хеш — от НЕсжатого логического блока, до сжатия.** Это (а) даёт короткое
   замыкание: на дубликате экономим CPU на сжатии/шифровании, а не только IOPS;
   (б) убирает баг «одинаковый plaintext сжался по-разному из-за другого уровня/словаря
   ZSTD → дедуп промахнулся». Так дедупит ZFS — по чексумме несжатого блока.

- **Сжатие** по recordsize (§4.5), не файл целиком: иначе для чтения хвоста 4-ГБ файла
  пришлось бы распаковывать всё с начала. ZSTD (жмёт сильнее) или LZ4 (распаковка почти
  на скорости RAM). В метаданных — сжатый и реальный размер.
- **Шифрование** AES-XTS: каждый кластер шифруется независимо, **tweak = адрес
  физического кластера**. Аппаратное AES-NI → почти нет потери скорости при
  параллельном расшифровывании. Расшифровка — после чтения, распаковка — после
  расшифровки.

### 5.2. Дедупликация по контент-хешу

```go
type DedupEntry struct {
    BlockAddr uint64 // где блок реально лежит на диске
    RefCount  uint32 // сколько файлов на него ссылаются
}
type DedupTable map[[32]byte]DedupEntry // ключ: BLAKE3 НЕсжатого логич. блока
```

Дедуп-хеш (BLAKE3 несжатого блока, для **совпадений**) и checksum (по шифртексту на
диске, для **bit-rot**, §B13) — это **два разных поля** с разными задачами. См. §B7
про verify и §B8 про взаимодействие с AES-XTS.

### 5.3. Полный конвейер записи блока

```
[Async-поток записи: логический блок recordsize (§4.5)]
                │
                ▼
   [Хеш BLAKE3 НЕсжатого блока]
                │
                ▼
   [Поиск в DedupTable (RAM)]
                │
     ┌──────────┴───────────┐
 (хеш НАЙДЕН)          (хеш НЕ НАЙДЕН)
     │                      │
[RefCount++]      [Сжатие ZSTD/LZ4 → напр. 11К]
     │                      │
     │              [Шифрование AES-XTS, tweak = адрес кластера]
     │                      │
     │              [Запись в N кластеров 4К]
     │                      │
     │              [DedupTable: addr, RefCount=1; Checksum шифртекста]
     └──────────┬───────────┘
                ▼
   [Запись экстента/адреса в карту блоков inode файла]
```

### 5.4. Удаление и счётчик ссылок

При `rm` у всех блоков файла `RefCount--`. Если `RefCount == 0` — блок больше никому
не нужен → передаётся сборщику мусора (§B5) на физическое освобождение. Если
`RefCount > 0` — блок остаётся, файл просто перестаёт на него ссылаться.

Эффект: скопировать `movie.mp4` десять раз → на диске он лежит **один раз**, 10 inode
ссылаются на одни блоки. Экономит место и ресурс SSD (write amplification).

### 5.5. Режимы сжатия и шифрования (опциональность) — РЕШЕНО

Ключевая асимметрия: **сжатие отменяемо поблочно, шифрование — нет.**

**Сжатие — пер-файловая опция, дефолт ВКЛ.**
- Управляется битом `FlagCompressed` в inode (§4.1). Для заведомо несжимаемых файлов
  (медиа, архивы, уже-зашифрованное извне) выключается пер-файлово — чтобы не тратить
  CPU на бесполезную ZSTD-попытку.
- Даже при «вкл» работает per-block raw-fallback (§B9): несжавшийся блок пишется raw,
  без раздувания. Поэтому «выкл» экономит **только CPU**, не место. Аналог
  `compression=on|off` на датасет в ZFS, дефолт on.

**Шифрование — опция на этапе `Format`, неизменная для пула, дефолт ВКЛ + opt-out.**
- Решается при создании пула и **фиксируется навсегда**. Пер-блочно/пер-файлово
  переключать нельзя по построению: keyring нужен до первой записи (§B4), смешение
  шифр/нешифр блоков ломает дедуп (§B8) и восстановление, смена режима = переписать
  все данные (XTS tweak привязан к адресу кластера).
- Дефолт — шифрованный пул; явный **opt-out** при `Format` создаёт plaintext-пул
  (bulk-данные, максимум скорости, нет риска «потерял ключ → потерял всё»).
- Режим хранится в uberblock полем `EncMode` (§cow/§B1); при `Mount` определяет, нужен
  ли keyring. `store` поддерживает **nil-cipher** путь для plaintext-пула.
- Согласовано с §B8 (дедуп в пределах одного ключа/пула) и §D-3 (кросс-юзерный дедуп
  выключен).

**Статус кода:** прототип сейчас hard-on по обоим (store всегда шифрует и жмёт с
raw-fallback). Реализация опциональности — задача интеграционной стадии Д: `EncMode` в
uberblock + nil-cipher в `store` + чтение `FlagCompressed` в конвейере.

# B. Что осталось за кадром (развёрнуто, с решениями)

Спроектирован «happy path». Ниже — узлы, без которых ФС повредит данные при первом же
сбое или нагрузке. Каждый пункт по шаблону **Проблема → Варианты → Рекомендация**.

### B1. Crash consistency — РЕШЕНО (Copy-on-Write)

- **Проблема:** запись группы изменений (блок данных + inode + bitmap + DedupTable) не
  атомарна — диск гарантирует атомарность только одного сектора. Сбой питания посередине
  оставляет несогласованное состояние. Худший случай — **перезапись на месте**: рваный
  блок (torn write), старое затёрто, новое не дописано.
- **Варианты:**
  (а) **WAL / журналирование** (как ext4) — пишем намерение в лог + fsync, потом
  применяем перезаписью на месте; после сбоя проигрываем/отбрасываем лог. Цена —
  двойная запись.
  (б) **Copy-on-Write** (как ZFS/Btrfs) — живые данные не трогаем; новую версию пишем
  в свободное место и атомарно (одной записью) переключаем корневой указатель.
  Промежуточного «рваного» состояния нет по построению. Цена — фрагментация, GC,
  «блуждающее дерево» (лечится батчингом транзакций).
- **Решение: CoW.** Выбор предопределён уже принятыми решениями:
  1. **Дедуп сделал блоки иммутабельными** — расшаренный блок (RefCount>1) физически
     нельзя перезаписать на месте. Мы уже наполовину CoW.
  2. **GC у нас и так есть** из-за дедупа/RefCount (§B5) — CoW не добавляет новой
     машинерии; для WAL пришлось бы держать GC *и* журнал.
  3. **Переменный recordsize + сжатие** (§4.5): изменённый кусок пережимается в другой
     размер → в старый слот не влезает → переаллокация неизбежна = CoW де-факто.
  4. **AES-XTS** (§B8): in-place-перезапись повторно использует адрес=tweak с новым
     plaintext (слабость XTS); CoW пишет в новый адрес → новый tweak.
  5. **Снапшоты бесплатно** (§B18): снапшот = удержанный старый root + запрет GC на
     достижимые блоки; ложится в модель RefCount.
  6. **Фрагментация — изначально «пофиг»** (NFR-4) + параллельное чтение по карте
     блоков (§4.4) её переживает.
- **Когда WAL был бы лучше (для протокола):** профиль «много мелких in-place
  перезаписей, без дедупа/сжатия, минимум фрагментации» (БД, ext4). Это **не** наш
  профиль — поэтому CoW.
- **Реализация:** батчинг изменений в транзакцию (txg-стиль, коммит раз в N мс) гасит
  «блуждающее дерево»; `fsync` (§B14) форсирует коммит текущей транзакции.
- **Конкретный механизм коммита (реализован и проверен на стадии Г).** Корень —
  **uberblock в двух чередующихся слотах** (блоки 0/1) с порядковым номером `Seq` и
  контрольной суммой BLAKE2b. Коммит: (1) снапшот метаданных (bitmap аллокатора + DDT)
  пишется в **неактивный** метаслот + `Sync`; (2) новый uberblock пишется в слот
  `Seq % 2` + `Sync` — это атомарный flip. Соседние `Seq` имеют разную чётность, поэтому
  новый uberblock физически не затирает слот с предыдущим коммитом. При монтировании
  читаются оба слота, выбирается старший валидный (магия+checksum); **рваная запись**
  отбраковывается контрольной суммой → откат к последнему согласованному коммиту.
  Уроки: торн-детект работает, только если **checksum покрывает значимую часть** блока;
  откат держится на **двухслотовости**. Crash-injection тесты (§B19) подтверждают откат
  при обрыве и в фазе метаданных, и в фазе uberblock.

### B2. Allocator / Free Space Map (учёт свободного места)

- **Проблема:** как за наносекунды найти свободный блок среди десятков/сотен миллионов?
- **Варианты:** bitmap свободных блоков; free-list; B+-дерево экстентов свободного
  места (как XFS).
- **Рекомендация:** **bitmap в RAM** + периодический сброс на диск (под защитой B1).
  Для параллелизма — **пер-сегментные/пер-зонные карты**, чтобы потоки аллоцировали из
  разных зон без общей блокировки. Кэш bitmap в памяти, аккуратная синхронизация с диском.

### B3. Восстановление RAM-структур при монтировании

- **Проблема:** DedupTable и фильтры Блума живут в RAM; перезагрузка их стирает.
- **Варианты:** (а) сериализовать на диск при чистом umount + быстрая загрузка;
  (б) полный пересчёт при старте; (в) держать дедуп-индекс **на диске** (как ZFS DDT)
  и кэшировать в RAM.
- **Рекомендация:** **on-disk дедуп-индекс (DDT-стиль)** как источник истины +
  снимок фильтров при чистом umount; при грязном размонтировании — пересчёт фильтров
  сегмента «по требованию» при первом открытии каталога (они дёшевы — §3.3).

### B4. Управление ключами (Key Management)

- **Проблема:** «шифруем AES-XTS» — а откуда мастер-ключ?
- **Варианты:** ключ из пароля напрямую (плохо); keyring-блок, обёрнутый ключом из пароля.
- **Рекомендация:** **«нулевой блок» / keyring**: хранит мастер-ключи (в т.ч. per-UID,
  §B8), сам зашифрован ключом, выведенным из пароля через **Argon2id** (память-зависимый;
  PBKDF2 — только как legacy-fallback). Ротация ключа = перешифровать keyring, а не весь
  диск.

### B5. Garbage Collection (сборка осиротевших блоков)

- **Проблема:** при `RefCount → 0` блок надо пометить свободным. Кто и когда?
- **Варианты:** синхронно в `rm` (тормозит пользователя); фоновый воркер.
- **Рекомендация:** **фоновый GC-воркер**. `rm` лишь декрементирует счётчики и ставит
  блок в очередь; воркер асинхронно метит блоки свободными в bitmap (§B2). При CoW
  (§B1) GC также собирает старые версии, на которые не ссылается ни один снапшот.

### B6. Race conditions / гранулярные блокировки

- **Проблема:** один поток читает файл, другой его удаляет/дописывает.
- **Варианты:** один глобальный mutex (убивает параллелизм); RWMutex на inode; RCU/lock-free.
- **Рекомендация:** **RWMutex на уровне отдельной inode** (а не на всю ФС). Критично —
  **единый порядок взятия блокировок** (родительский каталог → дочерний объект →
  inode), иначе дедлок. Для горячих read-only структур (фильтры) — atomic-указатели
  и подмена версии (§3.3).
- **Статус (реализовано в `fs`):** введён **FS-level `sync.RWMutex`** с контрактом
  «много читателей (`Lookup`/`Readdir`/`ReadFile`) / эксклюзивный писатель
  (`Mkdir`/`Create`/`WriteFile`/`Unlink`/`Commit`)». Это даёт параллельный Lookup,
  которого требует FUSE, и **проверено под `go test -race`** (16 горутин). Гранулярность
  пер-inode (для параллелизма *записи*) — будущая оптимизация; сейчас записи
  сериализуются. Конкурентный read-путь безопасен, т.к. кэшированный `dir.Directory`
  мутируется только под эксклюзивным локом (читатели не гонятся с писателем).

### B7. Коллизии хеша при дедупе — РЕШЕНО (доверять BLAKE3-256, без verify)

- **Проблема:** 64-битный xxhash для **дедупа** недопустим — коллизия означает, что
  два разных блока сольются в один → **тихая порча данных**. (Для ключей имён xxhash
  ок, потому что там есть финальная сверка строки.)
- **Варианты:** (а) доверять криптохешу без сверки (ZFS по умолчанию, SHA-256/Skein);
  (б) хеш + **байтовая сверка** перед слиянием (`verify`) — стоит одно лишнее чтение.
- **Решение: доверять BLAKE3-256 без verify.** Обоснование: при 256-битном
  криптостойком хеше вероятность коллизии (birthday bound ~2⁻¹²⁸) пренебрежимо мала —
  встретить её на практике менее вероятно, чем аппаратную ошибку памяти/диска, которые
  всё равно ловит checksum bit-rot (§B13). ZFS по той же логике доверяет SHA-256 по
  умолчанию. Хеш считается по **несжатому** логическому блоку (§5.1).
- **Запасной выход:** оставить `verify` доступным **флагом монтирования** (по
  умолчанию ВЫКЛ) — на случай паранойи/аудита, без переписывания формата.

### B8. AES-XTS vs дедупликация — РЕШЕНО (только внутрипуловый дедуп)

- **Проблема:** XTS использует **адрес блока как tweak** → одинаковый plaintext по
  разным адресам даёт **разный шифртекст**. Значит, дедуп на уровне шифртекста
  невозможен.
- **Разбор:** дедуп выполняется **до шифрования** — по BLAKE3 от *несжатого* блока
  (§5.1). Найден дубль → блок уже лежит на диске по адресу `A`, просто ссылаемся на `A`
  (физически он один, tweak один — консистентно). То есть **внутрипуловый дедуп
  (в пределах одного ключа) работает корректно**. Проблема была бы только у
  кросс-юзерного дедупа при разных ключах.
- **Решение: кросс-юзерный дедуп НЕ реализуется.** Единственный способ его получить —
  **convergent encryption** (ключ блока выводится из его же хеша), а это **дыра в
  безопасности**: открывает атаки confirmation-of-file (проверить, есть ли у юзера
  конкретный файл) и частотный анализ. Цена — потеря дедупа между пользователями с
  разными ключами — **принимается осознанно**. Дедуп остаётся в пределах одного
  пула/ключа. Convergent encryption из проекта **исключён** (не «выключен флагом» —
  вообще не закладывается).

### B9. Размер блока vs сжатие/дедуп — РЕШЕНО (переменный recordsize)

- **Проблема:** противоречие в исходном дизайне — для сжатия нужны крупные чанки, а
  единица аллокации = 4 КБ (мелкий блок плохо жмётся).
- **Решение (см. §4.5):** **разделить две гранулярности** — кластер аллокации 4 КБ +
  логический блок (recordsize) 4–32К степенями двойки, детерминированный от размера
  файла. recordsize — единица сжатия и дедупа.
- **Почему крупный recordsize НЕ ухудшает дедуп:** цель — **истинные дубликаты** (целые
  идентичные файлы: `cp`, повторная закачка), а не случайные совпадения блоков. У копий
  содержимое байт-в-байт и **одинаково выровнено** от смещения 0 → логические блоки
  совпадают при любом recordsize. Более того, крупный блок **выгоднее**: DedupTable в
  ~8× меньше записей (32К vs 4К) → меньше RAM и lookup'ов (а раздувание DDT — главная
  боль ZFS-дедупа, §B3). Детерминированный выбор recordsize от размера (§4.5) гарантирует,
  что две копии получают один recordsize и дедуплицируются.
- **Граница применимости (зафиксировать честно):** блочный дедуп с фиксированными
  границами ловит только **выровненные** дубликаты. Сдвинутые (B = «1 байт» + A → всё
  содержимое съехало) он **пропускает** — их берёт content-defined chunking (borg/restic).
  Это **осознанно вне области** BloomFS: цель — целые копии, не «похожие со сдвигом».

### B10. FUSE-потолок производительности

- **Проблема:** каждая операция через FUSE — это переключения контекста
  userspace↔ядро. 14 нс in-memory поиска утонут в микросекундах FUSE-оверхеда на
  сквозном пути.
- **Варианты:** смириться; FUSE writeback cache и большие read-ahead; FUSE
  passthrough / io_uring-интерфейс ядра (новее); в пределе — модуль ядра (отложено).
- **Рекомендация:** держать быстрый индекс (он бесплатен и помогает CPU/кэшу), но
  **бенчмаркать честно**: отдельно in-memory индекс, отдельно сквозную FUSE-операцию.
  Включить FUSE writeback cache и батчинг readdir/read. Не обещать «14 нс на открытие
  файла» — это латентность индекса, а не syscall.

### B11. Собственный page/buffer cache

- **Проблема:** `O_DIRECT` (§1.2) обходит кэш ОС → повторные чтения бьют по диску,
  расшифровка/распаковка повторяются.
- **Варианты:** не использовать O_DIRECT (двойное кэширование); свой кэш.
- **Рекомендация:** **собственный буферный кэш** (ARC или сегментированный LRU) уже
  **расшифрованных/распакованных** блоков. Это и есть смысл O_DIRECT — управлять
  кэшированием самим, а не дублировать кэш ядра. Кэш-инвалидация согласована с CoW
  (§B1) и дедупом.
- **Статус (частично реализовано):** в `fs` есть **кэш открытых каталогов** (`sync.Map`,
  inode→резидентный `dir.Directory`+inode) — это *метаданная* сторона кэша. Без него
  каждый `Lookup` перечитывал+расшифровывал+пересобирал весь каталог; с ним горячий
  Lookup — чистый RAM-хит (проверено: `TestDirCacheAvoidsReReads` — 0 лишних `ReadBlock`
  на 200 повторных Lookup). Осталось: кэш **блоков данных** (расшифрованных) + **вытеснение
  (LRU)** — сейчас кэш каталогов неограниченный (приемлемо для прототипа).

### B12. Readdir / листинг каталога

- **Проблема:** фильтр Блума отвечает только на «есть ли файл X». Для `ls` он
  бесполезен — нужен полный обход.
- **Варианты:** обход всех сегментов; отдельный отсортированный индекс имён.
- **Рекомендация:** `readdir` обходит **все сегменты** каталога и собирает записи
  (имя хранится в `DirectoryEntry`, §3.4). Определить порядок вывода (например, по
  `NameHash` — стабилен и дёшев) и поведение при конкурентной мутации (снимок списка
  сегментов под RLock, чтобы листинг был консистентным).

### B13. Целостность данных / bit-rot

- **Проблема:** диск может тихо испортить биты; дедуп-хеш считается до записи и не
  ловит порчу **после**.
- **Варианты:** без проверки; checksum на блок, сверяемый при чтении.
- **Рекомендация:** **отдельная контрольная сумма по тому, что лежит на диске
  (шифртекст)** — это **не** дедуп-хеш (тот по несжатому plaintext, §5.2). Дедуп-хеш
  ловит совпадения до записи, а bit-rot — порча уже после неё, поэтому нужны два разных
  поля. Для скорости checksum может быть усечённым BLAKE3 или CRC32C; хранится в inode
  (поле `Checksum`, §4.1) для инлайн-данных либо рядом с адресом блока в карте/DDT.
  Сверяется при чтении; при несовпадении + CoW (§B1) + снапшоты — восстановить старую
  версию или честно вернуть EIO, а не молча отдать мусор.

### B14. POSIX-семантика durability (fsync / барьеры)

- **Проблема:** приложения ждут, что после `fsync()` данные на диске. Нужны гарантии
  упорядочивания записи.
- **Варианты:** игнорировать (опасно); честный fsync с flush + барьерами.
- **Рекомендация:** реализовать `fsync`/`fdatasync` как принудительный сброс
  CoW-транзакции (или WAL) до точки согласованности + flush кэша устройства. Чётко
  задокументировать гарантии после сбоя (что переживёт, что нет).

### B15. Лимиты и переносимость on-disk формата

- **Проблема:** неуказанные лимиты и порядок байт ломают переносимость образа между
  машинами.
- **Рекомендация:** зафиксировать в суперблоке и документации: **endianness**
  (рекомендуется little-endian фиксированно), max размер файла (при `Size uint64` —
  16 EB), max длина имени, max длина пути, выравнивание структур (`#[repr(C)]` в Rust,
  явный padding в Go). Версия формата в суперблоке (§2) для будущих миграций.

### B16. xattrs, симлинки, спецфайлы, atime

- **Проблема:** реальные приложения требуют расширенных атрибутов, симлинков, иногда
  спецфайлов; `atime` при каждом чтении = write amplification.
- **Рекомендация:** заложить тип в inode (§4.1 уже есть Regular/Dir/Link); для
  **xattr** — отдельный блок/структура по указателю из inode (не пихать в 128 байт).
  Симлинки — короткий target inline, длинный — в блоке данных.
- **atime — РЕШЕНО: `relatime` по умолчанию** (§D-6). При CoW (§B1) обновление atime
  особенно дорого — это **новая запись блока** (write amplification + фрагментация +
  лишний мусор в снапшотах). Варианты: `strictatime` (POSIX-точно, но каждое чтение =
  запись — неприемлемо при CoW), `noatime` (максимум скорости, но ломает редкие
  приложения вроде mutt/tmpwatch), `relatime` (atime пишется только если он старше
  mtime/ctime или старше 24 ч). **Выбор — `relatime`**: дефолт современного Linux,
  даёт совместимость почти даром. `noatime` доступен флагом монтирования для
  read-heavy-сценариев, где atime вообще не нужен.

### B17. Квоты на пользователя

- **Проблема:** раз есть 256 UID, естественно ограничивать потребление места/инод.
- **Рекомендация:** таблица квот `UID → {использовано блоков, лимит, использовано инод,
  лимит}` (256 записей — мизер, всегда в RAM). Учёт при аллокации/освобождении (с
  поправкой на дедуп — считать ли shared-блок по числу ссылок). Простая фича, дающая
  большую ценность при многопользовательском сценарии.

### B18. Снапшоты (бонус CoW)

- **Проблема/возможность:** при CoW (§B1) снапшоты почти бесплатны.
- **Рекомендация:** снапшот = сохранённый корневой указатель + запрет GC на блоки,
  достижимые из него. RefCount дедупа (§5.2) уже учитывает множественные ссылки —
  снапшоты ложатся в ту же модель. Дать команды create/list/delete/rollback снапшота.

### B19. Стратегия тестирования

- **Проблема:** системный код тихо повреждает данные; обычные unit-тесты этого не ловят.
- **Рекомендация (многоуровнево):**
  - **Бенчмарки** `go test -bench=.` на 4000+ файлов (поиск hit/miss, перестройка фильтра).
  - **Property-тесты** инвариантов: число занятых блоков = Σ RefCount; после
    add+delete состояние = исходное; сериализация→десериализация = identity.
  - **Fuzzing** парсера on-disk структур (битый суперблок/inode не должен ронять процесс).
  - **Crash-injection**: эмулятор `BlockDevice`, обрывающий запись на случайном блоке
    (подход dm-flakey / CrashMonkey) → ФС обязана подняться в согласованное состояние.
  - **Stress/race**: `go test -race`, конкурентные read/write/delete по одному inode.

# D. Открытые вопросы для согласования (зафиксировать до кодинга)

1. ~~**Crash consistency: CoW или WAL?**~~ **РЕШЕНО:** Copy-on-Write — выбор
   предопределён иммутабельностью дедуп-блоков, переменным recordsize+сжатием и
   AES-XTS (§B1). Транзакции батчатся (txg-стиль).
2. ~~**Дедуп-хеш с verify или без?**~~ **РЕШЕНО:** доверять **BLAKE3-256 без verify**
   (коллизия ~2⁻¹²⁸ менее вероятна, чем сбой железа); `verify` остаётся флагом, выкл по
   умолчанию (§B7).
3. ~~**Кросс-юзерный дедуп: нужен ли?**~~ **РЕШЕНО: нет.** Convergent encryption —
   дыра в безопасности (confirmation-of-file), исключён из проекта; дедуп только в
   пределах одного ключа/пула (§B8).
4. ~~**Recordsize: фиксированный или переменный?**~~ **РЕШЕНО:** кластер 4 КБ +
   переменный логический блок 4–32К (степени двойки), детерминированный от размера
   файла (§4.5, §B9).
5. ~~**Размер inode: 64 или 128 байт?**~~ **РЕШЕНО:** 128 байт, 32 inode/блок, с
   инлайновыми экстентами и отдельным полем checksum (§4.1).
6. ~~**atime-политика по умолчанию?**~~ **РЕШЕНО:** `relatime` (дешёвый компромисс,
   критично при CoW); `noatime` — флагом монтирования (§B16).

**Все развилки §D разрешены.** Остаётся лишь тюнинг-константа: **пороги
размер→recordsize** (§4.5) — приняты дефолтными, при желании пересмотреть. Дальше —
реализация по стадиям §C.

# E. POSIX Compliance Matrix

Молодые ФС ломаются не на хранении данных, а на **несовпадении ожиданий ядра и
приложений**. Этот раздел — матрица семантических кейсов; каждый закрывается отдельным
тестом (`fs/posix_test.go`, далее — `tests/posix/` на mount), каждый найденный баг → новый
тест.

| # | Кейс | Семантика | Статус | Опора |
|---|------|-----------|--------|-------|
| E1 | rename | старый ИЛИ новый, никогда полфайла | **готово** (session) | CoW dir-data; имя↔inode (§4.2, §B1) |
| E2 | rename overwrite | атомарно заменить + освободить старую цель | **готово** (session) | RefCount (§5.4) |
| E3 | unlink открытого файла | каталог теряет имя, inode жив пока open | **готово** | `openCount` + `Nlink==0 && openCount==0 ⇒ reclaim` |
| E4 | hardlink | N имён → 1 inode | **готово** (РЕШЕНО: поддержать рано) | `Nlink` + имя↔inode |
| E5 | truncate grow/shrink, sparse | `truncate -s 1TB` без выделения террабайта | **готово** (changelog 1.14) | `Truncate` + sparse-рост через `WriteAt`>EOF (multi-extent) |
| E6 | mmap | страница меняется без `Write()` | **исключён в v1** (РЕШЕНО) | ломает read/write-модель |
| E7 | fsync durability | после fsync переживает power-loss | **готово** (committed durable + uncommitted исчезает) | CoW таблицы инод+bitmap+DDT (§B1); остаточная щель — deferred-free, см. ниже |
| E8 | append concurrency | 32 горутины, без потерь/дублей/перемешивания | **готово** (changelog 1.17) | атомарный `Append` под write-lock; стресс-тест `-race` |
| E9 | readdir во время мутации | консистентный снимок | **готово** (changelog 1.18) | `OpenDir` фиксирует снимок имён; `DirHandle.Next` пагинирует по нему |

Приоритет (по ревью): **E3, E1, E7** — если эти три верны, проект ведёт себя как настоящая
ФС, а не «интересное хранилище». E3/E1 закрыты на session-уровне и под `-race`.

## E-блокер №1: CoW таблицы инод — РЕШЕНО (changelog 1.12)

Реализация E1/E3/E4 вскрыла, что полная crash-атомарность метаданных и жёсткая гарантия E7
упирались в одно: таблица инод писалась **in-place** (вне CoW). Это закрыто:
- таблица инод — теперь RAM-структура `inode.Table`; `Get`/`Put` не трогают диск;
- на каждом commit она сериализуется в активный cow-метаслот как третья секция снапшота
  `[bitmap | DDT | inode-table]`, после чего атомарно переключается uberblock (§B1);
- отдельной in-place области инод больше нет (геометрия: два метаслота + data-регион);
- **итог:** uncommitted создание файла/имени не оставляет следа после сбоя
  (`TestUncommittedChangesVanish`); committed-состояние durable (`TestFsyncDurability`);
  rename/unlink/hardlink теперь крэш-атомарны как единый commit.

## E-блокер №2: отложенное освобождение кластеров (deferred-free) — РЕШЕНО (changelog 1.13)

Закрытие блокера №1 обнажило следующую щель в CoW: аллокатор освобождал кластеры в
RAM-bitmap **немедленно** (`bm.Free` в `store.Release`), и переиспользование внутри той же
незакоммиченной транзакции могло затереть на диске кластер, принадлежащий предыдущему
commit'у ⇒ порча состояния, к которому делается откат. Закрыто (§F1):
- `alloc.Bitmap.Defer` помечает освобождаемый прогон как **pending**, оставляя бит SET — до
  коммита прогон не переаллоцируется (pinned extent, как в Btrfs);
- `store.Release` теперь зовёт `Defer`, а не `Free` (немедленный `Free` остаётся только для
  отката неудачной `store.Write` — те кластеры выделены в этой же транзакции и не были
  закоммичены);
- `cow.Commit` после успешного flip uberblock вызывает `bm.ApplyDeferred()` → кластеры
  реально освобождаются и доступны со следующего commit'а (как deferred frees по txg в ZFS).
- **итог:** сбой до коммита оставляет освобождённые кластеры pinned (не затёрты), сбой во
  время коммита — тоже (ApplyDeferred идёт только после flip); `TestUncommittedChangesVanish`
  теперь покрывает и overwrite — закоммиченное состояние переживает откат.

# F. Путь до mount (что осталось перед FUSE)

Биндинг FUSE/WinFsp сам по себе — тонкий шим: он проецирует VFS-методы пакета `fs` в
обработчики ядра. Но чтобы mount был осмысленным, перед ним нужно закрыть несколько
блокеров. Стадия Д2 в §C расширяется этим разбиением.

| # | Пункт | Зачем до mount | Статус | Опора |
|---|-------|----------------|--------|-------|
| F1 | deferred-free кластеров | без него overwrite+crash портит committed-состояние | **готово** (changelog 1.13) | §E-блокер №2; alloc/store/cow |
| F2 | recordsize-блоки | поблочный (а не пофайловый) дедуп; локальные записи вместо переписи всего файла | **готово** (changelog 1.14) | §5, §4.4; решение по recordsize (§D) |
| F3 | multi-extent инод / внешний block-map | большой файл на фрагментированном диске; инлайн single-ref, дальше — внешний ref-list блок | **готово** (changelog 1.14) | §4.4 |
| F4 | addressable `ReadAt`/`WriteAt` | ядро шлёт `write(off,len)`; раньше API умел только файл целиком | **готово** (changelog 1.14) | §1.2; `ReadFile`/`WriteFile` остаются обёртками |
| F5 | free-list инод | bump-аллокатор течёт: reclaim лишь зануляет слот, id не переиспользуются | **готово** (changelog 1.15) | §B2; стек переиспользования, восстанавливается сканом при Mount |
| F6 | timestamps (a/m/ctime) | `getattr` иначе отдаёт epoch; поля в иноде есть, не заполняются | **готово** (changelog 1.16) | §4.1; clock-инъекция, noatime на чтении |
| F7 | E9 readdir-снимок при мутации | консистентный листинг во время записи | **готово** (changelog 1.18) | §B12, §E; `OpenDir`→`DirHandle` снимок |
| F8 | E8 стресс-тест append-конкурентности | 32 горутины без потерь/дублей | **готово** (changelog 1.17) | §B6, §E; атомарный `Append` + стресс-тест под `-race` |

Снято попутно: **F2–F4 закрыли E5** — `Truncate` (grow/shrink/в ноль) реализован, sparse-рост
через `WriteAt` за EOF (промежуток читается нулями). Остаётся **F5–F8**.

**Реализация F2–F4 (changelog 1.14):** содержимое файла режется на recordsize-записи (§4.5),
размер выводится из итогового размера файла и замораживается, как только файл получил данные.
Single-record файл хранит `store.Ref` инлайн в иноде (`FlagInlineExtents`); multi-record —
внешний ref-list блок-map (count u32 + N×`RefSize`), а инлайн лежит ссылка на него. Каждая
запись — независимая единица dedup/compress/encrypt, поэтому `ReadAt`/`WriteAt` трогают только
перекрытые записи; нетронутые записи переносятся со старыми ref (их dedup-refcount не дёргается),
а заменённые и старый контейнер block-map освобождаются через deferred-free (§F1).

**Реализация F5 (changelog 1.15):** `reclaim` зануляет слот, но переносит вперёд
**инкрементированную generation** (стейл-хэндл к старому файлу не алиасит переиспользованный
id) и кладёт id в RAM-стек переиспользования; `allocInode` сперва берёт из стека, иначе
bump. Стек не персистится — `Mount` восстанавливает его сканом закоммиченной таблицы на
слоты с `Nlink==0` (корень всегда `Nlink>=2`, не попадает), поэтому после краха он всегда
соответствует состоянию, к которому сделан откат. При reclaim каталога инвалидируется и
кэш каталогов (`dcache`), чтобы переиспользованный id не вернул устаревший снимок.

**Реализация F6 (changelog 1.16):** время (Unix-наносекунды) проставляется на мутациях через
инъектируемые часы (`clock`, поле — чтобы тесты гоняли детерминированно). Изменение
содержимого/размера (включая запись в каталог при create/unlink/link/rename) бьёт **mtime+ctime**
(`touchMod`, централизованно в `setData` + в `WriteAt`); изменение только метаданных — счётчик
ссылок, переименование — бьёт **ctime** (`touchMeta`). `atime` ставится при создании и записи, но
**не обновляется на чтении** (read-путь остаётся под `RLock` без мутаций — фактически `noatime`;
точную `relatime`-семантику добавит FUSE-слой). Времена входят в CoW-снимок и переживают remount.

**Реализация F8 (changelog 1.17):** добавлен атомарный `Append(id, data)` — читает размер и
дописывает под одним write-lock, поэтому конкурентные дозаписи не теряются, не дублируются и не
перемешиваются. Стресс-тест: 32 горутины × 50 записей, проверка «каждая запись ровно один раз,
ни одна не порвана» под `-race`. `WriteAt`/`Append` теперь делят общее `writeAtLocked`-ядро.

**Реализация F7 (changelog 1.18):** `OpenDir(id)` фиксирует снимок имён каталога (под RLock;
`dir.List` отдаёт свежий слайс, поэтому снимок заморожен), а `DirHandle.Next(n)` пагинирует по
нему. Конкурентные create/unlink/rename после открытия не добавляют, не убирают и не дублируют
имён в потоке — именно та стабильность, которой требует постраничный readdir ядра (`getdents`).

**Полностью закрыт §F.** Всё, что нужно перед осмысленным mount, готово; дальше — сам биндинг
(Стадия Д2): enforcement прав (или `default_permissions`), page cache, квоты, FUSE/WinFsp-шим.
Самый объёмный кусок (data-path F2–F4) сделан и даёт настоящий поблочный дедуп.

## Headline-бенч стадии Д2 (нужен mount)

`find` / `git status` / `git checkout` / `ripgrep` по **100 000 файлов в одном каталоге** —
здесь миллионы `lookup`/`stat`/`readdir`. Сегменты + Bloom + кэш каталогов либо сияют, либо
«FUSE всё съел». Это решающий тест основной идеи.
